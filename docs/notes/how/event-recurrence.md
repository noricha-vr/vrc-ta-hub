# 定期イベント生成パターン

## 固定日ルールは LLM に渡さない
- 問題: `毎月11日` のような単純な固定日ルールを `frequency=OTHER` のまま LLM 生成に委ねると、11日以外の日付が一度でも返ったときにその future event が次回生成の基準日を汚染し、定期実行のたびに日付ドリフトが増殖する。
- 解決: `毎月N日` のような固定日パターンはサーバー側で決定論的に解釈し、LLM を経由しない。
- 教訓: 自由記述ルールでも、機械的に解釈できるパターンは deterministic path に寄せたほうが安全。

## 壊れた future instance を基準日に使わない
- 問題: 定期イベント生成が `last_instance.date + 1日` を次回基準日に使っていると、誤生成された future instance が残っているだけでロジック修正後も自己回復しない。
- 解決: deterministic に解釈できるルールでは、future のルール外 instance を先に削除し、ルールに一致する最新 instance だけを基準日に使う。
- 教訓: 生成ロジック修正だけでは既存データ汚染は止まらないので、自己回復の cleanup 導線もセットで入れる。
