# Django migration パターン

## 新規テーブル migration 未適用でも主処理を 500 にしない
- 問題: 新しく追加したテーブルを `post_save` シグナルや非本質な副作用処理が即参照すると、デプロイ直後や migration 不整合の瞬間に `ProgrammingError` で本来の保存処理まで失敗する
- 解決:
  1. 保存の主目的ではない副作用処理は、対象テーブルの利用可否を先に確認する
  2. テーブル未作成なら warning を出して副作用だけスキップし、主処理は成功させる
  3. 例外を広く握りつぶすのではなく、テーブル未作成を明示的に判定する
  4. migration 適用自体は別途必ず実施する
- 教訓: migration 不整合で止めるべきなのは新機能や周辺機能であって、既存の承認・編集・保存フローまで巻き込まないほうが運用復旧しやすい
